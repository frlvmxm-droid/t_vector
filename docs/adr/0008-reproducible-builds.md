# ADR-0008: Reproducible builds via `uv.lock` + pinned Docker base

## Status
Accepted; shipped in Wave 8.2.

## Context
Three sources of dependency truth coexisted before Wave 8.2:

1. `requirements.txt` ‚Äî hand-edited, version *ranges* (`>=2.1`), used by
   `bootstrap_run.py` for the desktop launcher.
2. `requirements.lock` ‚Äî generated by `pip-compile` on a developer's
   machine, containing absolute paths from that machine
   (`/home/user/classification-tool/...`). Stale and not re-generated in CI.
3. `pyproject.toml` ‚Äî had no `[project.dependencies]` block at all, so
   `pip install .` was a no-op.

A nightly perf regression that turned out to be a `scikit-learn` 1.4 ‚Üí 1.5
silent change in `LinearSVC` calibration (Wave 6 incident #4) made the
"two CI runs from the same SHA can resolve to different wheels" risk
concrete. ADR-0005 (signed releases) also assumes the wheel set in a
release artefact matches what a consumer would re-download ‚Äî that
assumption needs a lock file to hold.

## Decision
1. **`uv.lock` is the source of truth** for fully-pinned, hashed
   wheels. Every dependency in the image, the test runner, and the
   release artefact resolves through it.
2. `pyproject.toml` declares the **direct** dependencies and the
   supported version *ranges* (`scikit-learn>=1.4`, `numpy>=1.24`, ‚Ä¶).
   Optional ML extras (`torch`, `transformers`, `sentence-transformers`,
   `setfit`, `umap-learn`, `rapidfuzz`, `datasketch`, `pymorphy3`) live
   under `[project.optional-dependencies]` so a minimal install (CLI,
   CI, Docker dev-image) does not pull a multi-GB wheel.
3. `requirements.txt` is **kept** for the `bootstrap_run.py` desktop
   launcher path ‚Äî it remains the only file end-users on Windows
   double-click into. It must stay in sync with `pyproject.toml`'s
   `dependencies` block; a CI smoke test enforces this (see ¬ß
   "Smoke test").
4. The legacy `requirements.lock` (pip-compile output with stale
   machine paths) is removed ‚Äî `uv.lock` supersedes it.
5. **Dockerfile** installs via `uv sync --frozen --extra dev` so the
   image build fails closed if `pyproject.toml` and `uv.lock` drift.
   The base image is pinned to `python:3.11.11-slim-bookworm` (explicit
   point release, not `:3.11`). Release builds SHOULD additionally pin
   the `@sha256:...` digest ‚Äî see "Pinning the base image" below.
6. Adding or bumping a dep is a two-step contract: edit
   `pyproject.toml`, then run `uv lock` and commit the resulting
   `uv.lock` in the same PR. CI verifies the lock is current via
   `uv lock --check` (added to `quality-gates.yml`).

## Pinning the base image
Tag-based pinning (`python:3.11.11-slim-bookworm`) protects against the
common case ‚Äî Python upstream re-tagging `python:3.11` after a point
release. It does **not** protect against base-image rebuilds that pick
up a new Debian security update mid-tag. For release builds (driven by
`RELEASE_CHECKLIST.md`), pin the digest explicitly:

```dockerfile
FROM python:3.11.11-slim-bookworm@sha256:<resolved-digest>
```

Resolve the current digest with `docker buildx imagetools inspect
python:3.11.11-slim-bookworm | grep Digest`. Bumping the digest is a
PR-visible event ‚Äî it shows up in `git diff Dockerfile`.

## Smoke test
`tests/test_repro_build.py`:

* Asserts `uv.lock` exists at the repo root and parses as TOML
  (uv writes `version = 1` at the top).
* Asserts every name in `requirements.txt` (the "–û–±—è–∑–∞—Ç–µ–ª—å–Ω—ã–µ" block,
  excluding comments and the optional-deps documentation) appears as
  a project dependency in `pyproject.toml`. Catches the
  `pyproject.toml` ‚Üî `requirements.txt` drift mode that motivated
  this ADR.
* Asserts `Dockerfile` calls `uv sync --frozen` (not `pip install
  <name>`), so a future "fix the build by adding pip install" diff
  fails review.

## Consequences
+ A CI run from any SHA resolves to the same wheel set, so a flaky test
  cannot be blamed on a transitive bump.
+ Release artefacts are bit-stable: same SHA ‚Üí same wheel set ‚Üí same
  sklearn / numpy / scipy ABI surface.
+ Docker layer caching stays effective because lock-file changes are
  the only thing that invalidates the dep layer.
‚àí Adding a dependency is now a two-file edit (pyproject + uv.lock),
  with mandatory `uv lock` step. Dev friction is real but small
  (‚âà 2 sec per resolve).
‚àí `uv.lock` is large (~580 KB, 3 100 lines) and will show up in PR
  diffs whenever a transitive dep moves. Reviewers should focus on
  the pyproject delta and trust the lock for transitives.
‚àí `bootstrap_run.py` still installs via `pip install -r
  requirements.txt`, so its install is *not* fully reproducible. That
  is acceptable ‚Äî it's a developer-onboarding convenience, not a
  release surface (cf. ADR-0005).
